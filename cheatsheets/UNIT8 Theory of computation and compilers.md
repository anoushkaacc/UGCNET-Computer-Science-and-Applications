## 1. Theory of Computation

### 1.1 Formal Language

- **Alphabet (Σ):** A finite non-empty set of symbols. Example: `Σ = {0, 1}`.
- **String (Word):** A finite sequence of symbols from Σ. The empty string is denoted `ε`.
- **Language (L):** A set of strings over Σ. `L ⊆ Σ*`.
- **Σ* (Kleene Star):** Set of all strings over Σ including ε.
- **Σ⁺:** `Σ* − {ε}` — all non-empty strings.
- **Length of string w:** `|w|`. `|ε| = 0`.
- **Concatenation:** `xy` = string x followed by string y. `|xy| = |x| + |y|`.
- **Reverse of w:** `w^R`. `(xy)^R = y^R x^R`.

**Operations on Languages:**
| Operation | Notation | Definition |
|---|---|---|
| Union | `L₁ ∪ L₂` | Strings in L₁ or L₂ |
| Concatenation | `L₁L₂` | `{xy | x ∈ L₁, y ∈ L₂}` |
| Kleene Star | `L*` | `{ε} ∪ L ∪ LL ∪ LLL ∪ ...` |
| Complement | `L'` or `¬L` | `Σ* − L` |
| Intersection | `L₁ ∩ L₂` | Strings in both L₁ and L₂ |

### 1.2 Non-Computational Problems

- Problems that cannot be solved by any algorithm (regardless of time/space).
- Also called **undecidable** or **non-recursive** problems.
- Proven via: Diagonal argument, reduction, Rice's theorem.

### 1.3 Diagonal Argument (Cantor's Diagonalisation)

- Proves that some sets are uncountable (have greater cardinality than the natural numbers).
- **Key result:** The set of all real numbers between 0 and 1 is uncountable.
- **Proof sketch:** Assume all reals can be listed as `r₁, r₂, r₃, ...`; construct a number d where the i-th decimal digit of d differs from the i-th digit of `rᵢ`. Then d is not in the list — contradiction.
- **Application to computation:** The set of all languages over Σ is uncountable, but the set of all Turing Machines is countable. Hence, there exist languages with no corresponding TM — **non-recursive languages**.

### 1.4 Russell's Paradox

- Let `S = {x | x ∉ x}` (the set of all sets that do not contain themselves).
- Question: Does `S ∈ S`?
  - If `S ∈ S` → by definition `S ∉ S`. Contradiction.
  - If `S ∉ S` → by definition `S ∈ S`. Contradiction.
- Shows that naive set theory is inconsistent; led to development of axiomatic set theory (ZFC).
- Analogous in computation: The **halting problem** is undecidable by a similar self-referential argument.

## 2. Regular Language Models

### 2.1 Deterministic Finite Automaton (DFA)

**Formal Definition:** A DFA is a 5-tuple `M = (Q, Σ, δ, q₀, F)` where:
- `Q` = finite set of states
- `Σ` = finite input alphabet
- `δ: Q × Σ → Q` = transition function (total function)
- `q₀ ∈ Q` = initial state
- `F ⊆ Q` = set of accepting (final) states

**Extended Transition Function:** `δ*: Q × Σ* → Q`
- `δ*(q, ε) = q`
- `δ*(q, wa) = δ(δ*(q, w), a)` for `w ∈ Σ*`, `a ∈ Σ`

**Language Accepted:** `L(M) = {w ∈ Σ* | δ*(q₀, w) ∈ F}`

**DFA Construction for Simple Languages:**
- `L = {w | w ends in 1}`: States: {q₀, q₁}; F = {q₁}; δ(q₀,0)=q₀, δ(q₀,1)=q₁, δ(q₁,0)=q₀, δ(q₁,1)=q₁.

### 2.2 Non-Deterministic Finite Automaton (NDFA / NFA)

**Formal Definition:** An NFA is a 5-tuple `M = (Q, Σ, δ, q₀, F)` where:
- `δ: Q × (Σ ∪ {ε}) → P(Q)` = transition function (returns a set of states)
- P(Q) = power set of Q

**ε-closure:** `ε-closure(q)` = set of states reachable from q using only ε-transitions.
```
ε-closure(q) = {q} ∪ {p | p reachable from q via ε-transitions}
ε-closure(S) = ∪_{q∈S} ε-closure(q)
```

**Extended Transition Function for NFA:**
- `δ*(q, ε) = ε-closure(q)`
- `δ*(q, wa) = ε-closure(∪_{p ∈ δ*(q,w)} δ(p, a))`

**Language Accepted:** `L(M) = {w | δ*(q₀, w) ∩ F ≠ ∅}`

### 2.3 Equivalence of DFA and NFA

**Theorem:** For every NFA N, there exists a DFA D such that `L(D) = L(N)`.

**Subset Construction (NFA → DFA):**
1. Start state of DFA = `ε-closure({q₀})`.
2. For each DFA state S (which is a subset of NFA states) and each symbol a:
   `δ_DFA(S, a) = ε-closure(∪_{q∈S} δ_NFA(q, a))`
3. A DFA state S is accepting if `S ∩ F_NFA ≠ ∅`.
4. DFA can have up to `2^|Q_NFA|` states.

**Key Points:**
- NFA and DFA accept exactly the same class of languages: **Regular Languages**.
- NFA may be more concise (exponentially fewer states) but DFA is deterministic.
- NFA with ε-transitions can be converted to NFA without ε-transitions, then to DFA.

### 2.4 Regular Languages

- A language is **regular** if it is accepted by some DFA/NFA.
- Equivalently: described by a regular expression or generated by a regular grammar.

### 2.5 Regular Grammars

A **Regular Grammar** is a grammar where all productions are of the form:
- **Right-Linear:** `A → aB` or `A → a` or `A → ε` (where A, B are non-terminals, a is terminal)
- **Left-Linear:** `A → Ba` or `A → a` or `A → ε`

**Equivalence:** Every regular language is generated by a right-linear (or left-linear) grammar and vice versa.

**Conversion (NFA → Right-Linear Grammar):**
- Non-terminals = states; start variable = start state.
- `δ(A, a) = B` → production `A → aB`.
- If `B ∈ F`, also add `A → a`.
- If `q₀ ∈ F`, add `A → ε`.

### 2.6 Regular Expressions (RE)

**Basis:**
- `∅` is a RE denoting the empty language.
- `ε` is a RE denoting `{ε}`.
- For each `a ∈ Σ`, `a` is a RE denoting `{a}`.

**Inductive Steps:** If r and s are REs denoting L(r) and L(s):
- `(r + s)` or `(r | s)` denotes `L(r) ∪ L(s)` — Union
- `(rs)` denotes `L(r)L(s)` — Concatenation
- `(r*)` denotes `(L(r))*` — Kleene Star
- `(r)` denotes `L(r)` — Grouping

**Operator Precedence (high to low):** `*` > concatenation > `+`

**Algebraic Laws:**
| Law | Expression |
|---|---|
| Commutativity of + | `r + s = s + r` |
| Associativity of + | `(r + s) + t = r + (s + t)` |
| Identity of + | `r + ∅ = r` |
| Identity of · | `εr = rε = r` |
| Annihilator | `∅r = r∅ = ∅` |
| Idempotent + | `r + r = r` |
| Distributive | `r(s+t) = rs + rt` |
| Kleene | `r* = ε + rr*` ; `(r*)* = r*` ; `(r+s)* = (r*s*)* = (r*+s*)*` |
| Arden's Lemma | If `L = QL ∪ R` and `ε ∉ Q`, then `L = Q*R` (unique solution) |

**Converting DFA to RE (State Elimination Method):**
1. Add new start state s and new accepting state f with ε-transitions.
2. Eliminate states one by one; update transitions with union/concatenation.
3. RE labelling remaining arc from s to f = answer.

**Converting RE to NFA (Thompson's Construction):**
- Build NFAs for basis cases; combine using union, concatenation, star NFA constructions.

### 2.7 Properties of Regular Languages (Closure Properties)

Regular languages are closed under:
- Union, Intersection, Complement, Concatenation, Kleene Star
- Difference (L₁ − L₂), Reversal (L^R)
- Homomorphism, Inverse Homomorphism
- Quotient: `L₁/L₂ = {x | ∃y ∈ L₂ such that xy ∈ L₁}`

**Decision Properties:**

| Problem | Decidable? | Method |
|---|---|---|
| Is `L(M) = ∅`? | Yes | Check if any accepting state is reachable from start |
| Is `w ∈ L(M)`? | Yes | Simulate DFA on w |
| Is `L(M)` finite? | Yes | Check for cycles reachable from start that reach accepting state |
| Is `L(M₁) = L(M₂)`? | Yes | Check L(M₁) △ L(M₂) = ∅ (symmetric difference) |
| Is `L(M₁) ⊆ L(M₂)`? | Yes | Check `L(M₁) − L(M₂) = ∅` |

### 2.8 Pumping Lemma for Regular Languages

**Statement:** If L is a regular language, then there exists a constant `p ≥ 1` (pumping length) such that for every string `w ∈ L` with `|w| ≥ p`, we can write `w = xyz` where:
1. `|y| ≥ 1` (y is non-empty)
2. `|xy| ≤ p` (xy is within first p characters)
3. `∀k ≥ 0: xy^k z ∈ L` (y can be pumped any number of times)

**Usage:** Used to prove a language is NOT regular (by contradiction).

**Proof Method:**
1. Assume L is regular with pumping length p.
2. Choose a specific string `w ∈ L` with `|w| ≥ p`.
3. Show that for every split `w = xyz` satisfying conditions 1 and 2, some `xy^k z ∉ L`.
4. Contradiction — L is not regular.

**Example:** `L = {0^n 1^n | n ≥ 0}` is not regular.
- Choose `w = 0^p 1^p`.
- By conditions: `xy = 0^j` (j ≤ p), so `y = 0^i` (i ≥ 1).
- `xy²z = 0^(p+i) 1^p ∉ L` (unequal number of 0s and 1s).

### 2.9 Non-Regular Languages

- `{0^n 1^n | n ≥ 0}`, `{ww^R | w ∈ {0,1}*}`, `{0^(n²) | n ≥ 0}` are not regular.
- Proven using pumping lemma or closure properties.

### 2.10 Lexical Analysis

- First phase of compiler; converts source code to token stream.
- **Tokens:** Identifiers, keywords, literals, operators, punctuation.
- **Lexeme:** Actual string matched.
- **Pattern:** Regular expression describing the token class.
- Implemented using **DFA** (compiled from regular expressions via NFA).
- Tools: **Lex / Flex** — take RE specifications and generate lexical analyser code.

**Lexical Analysis Steps:**
```
Source Program → [Scanner (DFA)] → Token Stream → [Parser]
RE specifications → Thompson's Construction → NFA → Subset Construction → DFA → Minimise DFA → Scanner
```

**DFA Minimisation (Table-Filling / Myhill-Nerode):**
- Mark all pairs (p, q) where one is accepting and one is not.
- Iteratively mark (p, q) if δ(p, a) and δ(q, a) are already marked for some a.
- Unmarked pairs → equivalent states → merge.
- **Myhill-Nerode Theorem:** A language L is regular if and only if the number of equivalence classes of `≡_L` (right invariance relation) is finite. The minimum DFA has exactly as many states as equivalence classes.

## 3. Context-Free Languages

### 3.1 Context-Free Grammar (CFG)

**Formal Definition:** A CFG is a 4-tuple `G = (V, T, P, S)` where:
- `V` = finite set of variables (non-terminals)
- `T` = finite set of terminals; `V ∩ T = ∅`
- `P` = finite set of productions of the form `A → α` where `A ∈ V`, `α ∈ (V ∪ T)*`
- `S ∈ V` = start symbol

**Derivation:** `α ⇒ β` (α derives β in one step by applying a production).
- `⇒*` = derives in zero or more steps.
- **Leftmost Derivation (lmd):** Always expand the leftmost non-terminal.
- **Rightmost Derivation (rmd):** Always expand the rightmost non-terminal.

**Language Generated:** `L(G) = {w ∈ T* | S ⇒* w}`

**Sentential Form:** Any string α such that `S ⇒* α` (may contain variables and terminals).

### 3.2 Simplification of CFGs

**Useless Symbols:** A symbol X is useless if it is not reachable from S or does not derive any terminal string.
- **Step 1:** Remove non-generating symbols (variables that derive no terminal string).
- **Step 2:** Remove non-reachable symbols (not reachable from S).

**ε-Productions:** Productions of the form `A → ε`.
- **Nullable:** A variable A is nullable if `A ⇒* ε`.
- To eliminate: For each production, add versions with and without nullable variables on RHS; remove `A → ε` (unless A = S and ε ∈ L(G)).

**Unit Productions:** Productions of the form `A → B` (single variable).
- Find unit pairs: (A, B) means `A ⇒* B` via unit productions.
- Replace `A → B` with all productions `A → α` for each `B → α` (non-unit).

### 3.3 Chomsky Normal Form (CNF)

**Definition:** A CFG is in CNF if every production is of the form:
- `A → BC` (two non-terminals), or
- `A → a` (single terminal), or
- `S → ε` (only if ε ∈ L(G))

**Conversion to CNF:**
1. Eliminate ε-productions.
2. Eliminate unit productions.
3. Eliminate useless symbols.
4. For each terminal a in a body of length ≥ 2: introduce new variable `Tₐ → a`, replace a with Tₐ.
5. For bodies of length ≥ 3: break into binary rules using new variables.
   - `A → B₁B₂...Bₖ` (k ≥ 3) → `A → B₁C₁`, `C₁ → B₂C₂`, ..., `C_{k-2} → B_{k-1}Bₖ`

**Use of CNF:** CYK (Cocke-Younger-Kasami) parsing algorithm requires CNF.

**CYK Algorithm:**
- DP algorithm; fills an n×n triangular table `T[i][j]` = set of variables that derive `wᵢwᵢ₊₁...wⱼ`.
- `T[i][i] = {A | A → wᵢ ∈ P}`
- `T[i][j] = {A | A → BC, B ∈ T[i][k], C ∈ T[k+1][j] for some i ≤ k < j}`
- `w ∈ L(G)` iff `S ∈ T[1][n]`
- Time complexity: O(n³ × |G|)

### 3.4 Greibach Normal Form (GNF)

**Definition:** A CFG is in GNF if every production is of the form:
- `A → aα` where `a ∈ T` and `α ∈ V*` (body starts with a terminal)

**Significance:**
- Every CFL (without ε) has a GNF grammar.
- GNF grammars are directly simulated by PDAs where each step reads one input symbol.
- Derivation of string of length n requires exactly n production steps.
- Eliminates left recursion naturally.

**Conversion:** Remove left recursion → Substitute to ensure each production starts with terminal.

### 3.5 Ambiguity

- A CFG is **ambiguous** if some string has two or more distinct **parse trees** (equivalently, two distinct leftmost derivations).
- A CFL is **inherently ambiguous** if every CFG for it is ambiguous.
- Example of ambiguous grammar: `E → E + E | E * E | id` (no precedence/associativity).
- **Unambiguous equivalent:** `E → E + T | T`, `T → T * F | F`, `F → id` (enforces precedence and left-associativity).

**Ambiguity is undecidable** — there is no algorithm to determine if an arbitrary CFG is ambiguous.

### 3.6 Parse Tree

- A tree representing a derivation; root = start symbol, internal nodes = variables, leaves = terminals/ε.
- Yield of a parse tree = string of terminals at leaves (left to right) = derived string.
- One parse tree ↔ one leftmost derivation ↔ one rightmost derivation (in unambiguous grammar).

### 3.7 Pushdown Automaton (PDA)

**Formal Definition:** A PDA is a 7-tuple `P = (Q, Σ, Γ, δ, q₀, Z₀, F)` where:
- `Q` = finite set of states
- `Σ` = input alphabet
- `Γ` = stack alphabet
- `δ: Q × (Σ ∪ {ε}) × Γ → P(Q × Γ*)` = transition function
- `q₀` = start state
- `Z₀ ∈ Γ` = initial stack symbol (bottom-of-stack marker)
- `F ⊆ Q` = accepting states

**Configuration (Instantaneous Description):** `(q, w, γ)` — state, remaining input, stack contents.

**Transition:** `(q, aw, Zγ) ⊢ (p, w, αγ)` if `(p, α) ∈ δ(q, a, Z)`.

**Acceptance:**
- **By Final State:** `L(P) = {w | (q₀, w, Z₀) ⊢* (q, ε, γ) for some q ∈ F, γ ∈ Γ*}`
- **By Empty Stack:** `N(P) = {w | (q₀, w, Z₀) ⊢* (q, ε, ε) for any q ∈ Q}`

Both acceptance modes are equivalent — any language accepted by one can be accepted by the other.

**Non-Deterministic PDA (NPDA):** δ may return multiple (state, stack) pairs. NPDAs are strictly more powerful than DPDAs.

**Deterministic PDA (DPDA):**
- `|δ(q, a, Z)| + |δ(q, ε, Z)| ≤ 1` for all q, a, Z.
- Accepts **Deterministic CFLs (DCFLs)**, a proper subset of CFLs.

### 3.8 Equivalence of PDAs and CFGs

**Theorem:** The class of languages accepted by PDAs (by empty stack or final state) is exactly the class of context-free languages.

**CFG → PDA (construction):**
1. Single state q; stack-based simulation of leftmost derivation.
2. If top of stack is variable A: guess production A → α, replace A with α.
3. If top of stack is terminal a: match with input symbol a, pop.
4. Accept when stack is empty and input is exhausted.

**PDA → CFG (construction):**
- Create variable `[pAq]` representing: PDA goes from state p to state q while popping symbol A.
- Productions derived from transitions of the PDA.

### 3.9 Properties of Context-Free Languages

**Pumping Lemma for CFLs:**

**Statement:** If L is a CFL, there exists a pumping length `p` such that for every string `w ∈ L` with `|w| ≥ p`, we can write `w = uvxyz` where:
1. `|vxy| ≤ p`
2. `|vy| ≥ 1` (v or y is non-empty)
3. `∀k ≥ 0: uv^k xy^k z ∈ L`

**Closure Properties of CFLs:**

| Operation | Closed? |
|---|---|
| Union | Yes |
| Concatenation | Yes |
| Kleene Star | Yes |
| Intersection | No (not closed) |
| Complement | No (not closed) |
| Intersection with Regular | Yes |
| Homomorphism | Yes |
| Inverse Homomorphism | Yes |
| Difference with Regular | Yes |

**Key Non-Examples (proven using pumping lemma):**
- `{a^n b^n c^n | n ≥ 1}` is not CFL.
- `{ww | w ∈ {a,b}*}` is not CFL.

**Decision Properties:**

| Problem | Decidable? |
|---|---|
| Is `w ∈ L(G)`? | Yes (CYK algorithm) |
| Is `L(G) = ∅`? | Yes |
| Is `L(G)` finite? | Yes |
| Is `L(G) = Σ*`? | Undecidable |
| Is `L(G₁) = L(G₂)`? | Undecidable |
| Is `L(G₁) ∩ L(G₂) = ∅`? | Undecidable |
| Is G ambiguous? | Undecidable |

## 4. Turing Machines

### 4.1 Standard Turing Machine (TM)

**Formal Definition:** A TM is a 7-tuple `M = (Q, Σ, Γ, δ, q₀, B, F)` where:
- `Q` = finite set of states
- `Σ` = input alphabet; `Σ ⊆ Γ`
- `Γ` = tape alphabet; `Σ ⊂ Γ`
- `δ: Q × Γ → Q × Γ × {L, R}` = transition function
- `q₀ ∈ Q` = start state
- `B ∈ Γ − Σ` = blank symbol
- `F ⊆ Q` = accepting states

**Configuration:** `α q β` — tape content left of head, current state, tape content from head onwards.

**Move:** `δ(q, X) = (p, Y, D)` means: in state q, reading X → write Y, move in direction D (L or R), go to state p.

**Acceptance:** M accepts w if starting in `q₀ w`, it eventually enters a state in F. M halts by entering F or by getting stuck (no transition defined).

**Language Accepted:** `L(M) = {w ∈ Σ* | M accepts w}`

### 4.2 Variations of Turing Machines

All the following are equivalent in computational power to the standard TM:

| Variation | Description |
|---|---|
| **Multi-tape TM** | k tapes with independent heads; simulated by 1-tape TM with O(t²) slowdown |
| **Non-deterministic TM (NTM)** | Multiple choices at each step; simulated by deterministic TM |
| **Semi-infinite Tape TM** | Tape extends only to the right; equivalent by encoding left part |
| **Multi-dimensional TM** | 2D or k-dimensional tape; equivalent to 1D TM |
| **Enumerator** | Generates all strings in a language; equivalent to TM |

**Multi-tape to 1-tape simulation:** Use single tape with k tracks separated by special markers.

### 4.3 Universal Turing Machine (UTM)

- A TM that can simulate any other TM.
- Input: Encoding `⟨M, w⟩` (description of TM M + input w).
- The UTM simulates M on w.
- **Encoding:** TM M → binary string `⟨M⟩` (via Gödel numbering).
- Existence of UTM is the theoretical basis for the **stored-program computer**.

### 4.4 Church-Turing Thesis

- **Statement:** Any function that can be effectively computed by any computational model is computable by a Turing Machine.
- Not a mathematical theorem (cannot be proved); a thesis/hypothesis.
- Supported by: equivalence of TM, lambda calculus, recursive functions, RAM model, etc.
- **Models of Computation:** TM, Lambda Calculus (Church), Recursive Functions (Gödel-Kleene), Post Systems — all compute the same class of functions.

### 4.5 Recursive and Recursively Enumerable Languages

| Class | Recogniser | Description |
|---|---|---|
| **Recursive (Decidable)** | TM that always halts | TM accepts strings in L; rejects strings not in L; always terminates |
| **Recursively Enumerable (RE)** | TM that may loop | TM accepts strings in L; may loop forever on strings not in L |
| **Non-RE** | No TM | No Turing Machine can recognise the language |

**Relationships:**
- Every recursive language is RE.
- There exist RE languages that are not recursive (e.g., the halting language).
- If L and L' are both RE, then L is recursive.

**Closure Properties:**

| Operation | Recursive | RE |
|---|---|---|
| Union | Closed | Closed |
| Intersection | Closed | Closed |
| Complement | Closed | Not closed |
| Concatenation | Closed | Closed |
| Kleene Star | Closed | Closed |

### 4.6 Context-Sensitive Languages and Unrestricted Grammars

**Context-Sensitive Grammar (CSG):**
- Productions: `αAβ → αγβ` where A ∈ V, α, β, γ ∈ (V∪T)*, |γ| ≥ 1.
- Every production is non-length-decreasing (except possibly S → ε).
- Recognised by **Linear Bounded Automata (LBA)** — TM where tape is bounded by input length.

**Unrestricted Grammar:**
- Productions: `α → β` where α ∈ (V∪T)* contains at least one variable.
- No restrictions on form; generates all RE languages.
- Recognised by Turing Machines.

### 4.7 Chomsky Hierarchy

| Type | Grammar | Language Class | Automaton |
|---|---|---|---|
| **Type 0** | Unrestricted | Recursively Enumerable | Turing Machine |
| **Type 1** | Context-Sensitive (CSG) | Context-Sensitive (CSL) | Linear Bounded Automaton (LBA) |
| **Type 2** | Context-Free (CFG) | Context-Free (CFL) | Pushdown Automaton (PDA) |
| **Type 3** | Regular (RG) | Regular | Finite Automaton (DFA/NFA) |

**Proper Inclusions:** Regular ⊂ CFL ⊂ CSL ⊂ RE

### 4.8 TM Constructions for Simple Problems

**TM for `{a^n b^n | n ≥ 1}`:**
```
1. Scan right; mark one 'a' with X.
2. Continue right; mark one 'b' with Y.
3. Scan left to leftmost unmarked 'a'; repeat.
4. If all a's marked, check no unmarked b's remain; accept.
5. If mismatch → reject.
```

**TM for addition:** Mark n ones, then m ones; result = n+m ones on tape.

## 5. Unsolvable Problems and Computational Complexity

### 5.1 Halting Problem

**Statement:** Given an encoding `⟨M, w⟩` of TM M and input w, determine whether M halts on input w.

**Halting Language:** `H = {⟨M, w⟩ | M is a TM and M halts on input w}`

**Theorem:** H is RE but not recursive (not decidable).

**Proof (by contradiction using diagonalisation):**
1. Assume TM H_decide decides the halting problem.
2. Construct TM D: on input `⟨M⟩`, run H_decide on `⟨M, ⟨M⟩⟩`. If H_decide says "halts" → loop forever; if "loops" → accept.
3. Run D on `⟨D⟩`: If D halts → D loops (contradiction); if D loops → D halts (contradiction).
4. Therefore, H_decide cannot exist.

### 5.2 Post Correspondence Problem (PCP)

**Instance:** A finite list of pairs of strings `{(u₁, v₁), (u₂, v₂), ..., (uₖ, vₖ)}` over some alphabet.

**Question:** Is there a sequence of indices `i₁, i₂, ..., iₘ` (m ≥ 1, indices may repeat) such that `u_{i₁}u_{i₂}...u_{iₘ} = v_{i₁}v_{i₂}...v_{iₘ}`?

**Theorem:** PCP is undecidable.
- Proven by reducing the halting problem (or acceptance problem for TMs) to PCP.
- **Modified PCP (MPCP):** Like PCP but first pair must be `(u₁, v₁)`.
- Reduction: Halting problem ≤_m MPCP ≤_m PCP.

**Use:** PCP used to prove undecidability of problems about CFGs (e.g., ambiguity, intersection non-emptiness).

### 5.3 Undecidable Problems for Context-Free Languages

| Problem | Decidable? |
|---|---|
| Is G ambiguous? | Undecidable |
| Is `L(G) = Σ*`? | Undecidable |
| Is `L(G₁) = L(G₂)`? | Undecidable |
| Is `L(G₁) ∩ L(G₂) = ∅`? | Undecidable |
| Is `L(G₁) ⊆ L(G₂)`? | Undecidable |
| Is `L(G)` regular? | Undecidable |

### 5.4 Measuring and Classifying Complexity

**Time Complexity:** `T_M(w)` = number of steps TM M takes on input w.
`T_M(n) = max{T_M(w) | |w| = n}` (worst case).

**Space Complexity:** `S_M(n)` = maximum number of tape cells used on any input of length n.

**Complexity Classes:**

| Class | Definition |
|---|---|
| **P** | Problems decidable in polynomial time: `∃k: T(n) = O(n^k)` |
| **NP** | Problems verifiable in polynomial time (or solvable by NTM in polynomial time) |
| **co-NP** | Complement of NP problems (e.g., TAUTOLOGY) |
| **PSPACE** | Solvable in polynomial space |
| **EXPTIME** | Solvable in exponential time |
| **L** | Solvable in O(log n) space |
| **NL** | Non-deterministically solvable in O(log n) space |

**Relationships:**
`P ⊆ NP ∩ co-NP`, `NP ⊆ PSPACE ⊆ EXPTIME`
- `P ≠ EXPTIME` (proven).
- `P = NP`? — open question (Millennium Prize Problem).

### 5.5 Tractable and Intractable Problems

- **Tractable:** Solvable in polynomial time (class P). Example: Sorting, BFS, MST, shortest path.
- **Intractable:** No known polynomial-time algorithm. Example: TSP, graph colouring, SAT.

**NP-Complete Problems:**
- A problem X is NP-complete if:
  1. X ∈ NP.
  2. Every problem in NP is polynomial-time reducible to X (NP-hard).
- **Reduction `A ≤_p B`:** If B is solvable in polynomial time, so is A.

**Key NP-Complete Problems:**

| Problem | Description |
|---|---|
| **SAT (Boolean Satisfiability)** | Is a Boolean formula satisfiable? (Cook-Levin theorem — first NP-complete) |
| **3-SAT** | SAT where each clause has exactly 3 literals |
| **Clique** | Does graph G have a clique of size k? |
| **Vertex Cover** | Does G have a vertex cover of size ≤ k? |
| **Independent Set** | Does G have an independent set of size ≥ k? |
| **Hamiltonian Circuit** | Does G have a Hamiltonian circuit? |
| **TSP** | Is there a tour of cost ≤ k? |
| **Graph Coloring** | Can G be coloured with k colours? (k ≥ 3) |
| **Subset Sum** | Is there a subset summing to target? |

**Cook-Levin Theorem:** SAT is NP-complete.
**Reductions:** 3-SAT ≤_p Clique ≤_p Independent Set ≤_p Vertex Cover.

## 6. Syntax Analysis

### 6.1 Associativity and Precedence

- **Precedence:** `*` has higher precedence than `+`.
- **Associativity:** Left-associative: `a - b - c = (a - b) - c`; Right-associative: `a ^ b ^ c = a ^ (b ^ c)`.
- Enforced by **grammar structure** — separate non-terminals for each precedence level.

### 6.2 Grammar Transformations

**Left Recursion Elimination:**
- **Direct left recursion:** `A → Aα | β` → Replace with `A → βA'`, `A' → αA' | ε`.
- **Indirect left recursion:** Substitute rules to find and eliminate.

**Left Factoring:**
- If `A → αβ | αγ` → Factor: `A → αA'`, `A' → β | γ`.
- Required for predictive (top-down) parsing; eliminates backtracking.

### 6.3 First and Follow Sets

**FIRST(α):** Set of terminals that begin strings derived from α.
- `FIRST(a) = {a}` for terminal a.
- `FIRST(ε) = {ε}`.
- `FIRST(A) = ∪ FIRST(α)` for each production `A → α`.
- If `A → ε` then `ε ∈ FIRST(A)`.
- `FIRST(X₁X₂...Xₖ)`: Add `FIRST(X₁) − {ε}`; if `ε ∈ FIRST(X₁)`, add `FIRST(X₂) − {ε}`; etc.

**FOLLOW(A):** Set of terminals that can appear immediately after A in some sentential form.
- `$ ∈ FOLLOW(S)` ($ = end of input).
- If `B → αAβ`: Add `FIRST(β) − {ε}` to `FOLLOW(A)`.
- If `B → αA` or `B → αAβ` where `ε ∈ FIRST(β)`: Add `FOLLOW(B)` to `FOLLOW(A)`.

### 6.4 Top-Down Parsing

- Builds parse tree from root (start symbol) to leaves.
- At each step, predict which production to use for current non-terminal.

**Recursive Descent Parsing:**
- Each non-terminal corresponds to a procedure.
- May require backtracking; not efficient for all grammars.

**Predictive Parsing (LL(1)):**
- No backtracking; uses FIRST and FOLLOW to decide production.
- Requires grammar to be LL(1) — no ambiguity, no left recursion, left-factored.

**LL(1) Parsing Table Construction:**
- For each production `A → α`:
  - For each terminal `a ∈ FIRST(α)`: Add `A → α` to `M[A, a]`.
  - If `ε ∈ FIRST(α)`: For each terminal `b ∈ FOLLOW(A)`: Add `A → α` to `M[A, b]`.
  - If `ε ∈ FIRST(α)` and `$ ∈ FOLLOW(A)`: Add `A → α` to `M[A, $]`.
- Grammar is LL(1) iff no entry has multiple productions.

**LL(1) Parser Operation:**
- Stack contains grammar symbols + `$` (bottom).
- If top of stack = current input = terminal: pop stack, advance input.
- If top = non-terminal A, input = a: apply production from M[A, a]; replace A with production body (right-to-left).
- If top = `$` and input = `$`: accept.

### 6.5 Bottom-Up Parsing (Shift-Reduce)

- Builds parse tree from leaves to root.
- Recognises **handles** — rightmost non-terminal's production body.

**Operations:**
- **Shift:** Push next input symbol onto stack.
- **Reduce:** Replace top of stack (a handle) with corresponding non-terminal.
- **Accept:** When stack = `$S` and input = `$`.
- **Error:** Parsing error.

**Viable Prefix:** A prefix of a right-sentential form that can appear on the stack of a shift-reduce parser.

### 6.6 LR Parsing

- **LR(k):** Left-to-right scan, Rightmost derivation in reverse, k tokens lookahead.
- Most powerful class of parsers for deterministic CFLs.

**LR Parser Components:**
- **Stack** (state + symbol pairs), **Input**, **Action Table**, **Goto Table**.

**Action Table Entry:**
- `shift j`: Push state j onto stack.
- `reduce A → β`: Pop |β| states; push goto[top_state, A].
- `accept`.
- `error`.

**Types of LR Parsers:**

| Parser | States | Power | Notes |
|---|---|---|---|
| **LR(0)** | Fewest | Weakest | No lookahead; rarely sufficient |
| **SLR(1)** | Same as LR(0) | More powerful | Uses FOLLOW sets for reduce decisions |
| **Canonical LR(1)** | Most | Most powerful | Full 1-token lookahead; large tables |
| **LALR(1)** | Same as SLR | Between SLR and CLR(1) | Merge LR(1) states with same LR(0) core; practical |

**LR(0) Item:** A production with a dot indicating how much has been parsed. `A → α•β` (dot before β).
- **Kernel Items:** Items with dot not at left end (except initial item `S' → •S`).
- **Closure Operation:**
  ```
  closure({A → α•Bβ}) += {B → •γ | B → γ ∈ P}
  ```
- **Goto Operation:**
  ```
  goto(I, X) = closure({A → αX•β | A → α•Xβ ∈ I})
  ```
- **Canonical Collection of LR(0) Items:** All states reachable by goto.

**SLR Reduce Condition:** Reduce `A → α` in state s if:
- `A → α•` ∈ items(s), and
- current input `a ∈ FOLLOW(A)`.

**LR(1) Item:** `[A → α•β, a]` — includes lookahead symbol a.
**LALR(1):** Merge all LR(1) states with the same LR(0) core (first component); use combined lookaheads.
- LALR(1) has same number of states as LR(0)/SLR.
- If merging creates reduce-reduce conflict → grammar is not LALR(1).

## 7. Semantic Analysis

### 7.1 Attribute Grammar

- A CFG augmented with **attributes** (values) associated with grammar symbols.
- **Synthesized Attribute:** Value computed from attributes of children and current node.
- **Inherited Attribute:** Value computed from attributes of parent and/or siblings.

**Formal Definition:** Attribute grammar is a triple `(G, A, R)`:
- G = CFG
- A = set of attributes for each symbol
- R = set of **semantic rules** (one per attribute per production)

### 7.2 Syntax-Directed Definitions (SDDs)

- Associate semantic rules with each production of the grammar.
- **Annotated Parse Tree:** Parse tree with attribute values at each node.

**Example — Arithmetic Expression:**
```
E → E₁ + T    { E.val = E₁.val + T.val }
E → T          { E.val = T.val }
T → T₁ * F    { T.val = T₁.val * F.val }
T → F          { T.val = F.val }
F → (E)        { F.val = E.val }
F → digit      { F.val = digit.lexval }
```

**S-Attributed SDD:** Uses only synthesized attributes.
- Can be evaluated in a single bottom-up pass (left-to-right, postorder).

**L-Attributed SDD:** Attributes evaluated in a left-to-right DFS order.
- For each production `A → X₁X₂...Xₙ`:
  - Each synthesized attribute of A may depend on any attribute of Xᵢ.
  - Each inherited attribute of Xᵢ may depend only on inherited attributes of A and attributes of X₁, X₂, ..., Xᵢ₋₁.
- S-attributed ⊂ L-attributed.

### 7.3 Dependency Graph and Evaluation Order

**Dependency Graph:**
- Node for each attribute instance.
- Edge `b → c` if attribute c depends on attribute b.
- **Topological sort** of dependency graph gives evaluation order.
- If dependency graph has a cycle → attribute grammar has no well-defined semantics.

**Evaluation Orders:**
- **Post-order (bottom-up):** For S-attributed; evaluate children before parent.
- **Pre-order (top-down):** Can evaluate some inherited attributes.
- **Multiple passes:** When topological order cannot be determined statically.

### 7.4 Type Checking

- Verify that operations are applied to compatible types.
- **Type System:** Set of rules for assigning type expressions to constructs.
- **Type Expression:** Describes a type. Composed from: basic types, type constructors (array, pointer, function, struct).
  - `array(n, T)`: array of n elements of type T.
  - `T₁ → T₂`: function from T₁ to T₂.
  - `pointer(T)`: pointer to T.

**Type Equivalence:**
- **Structural Equivalence:** Two types are equivalent if they have the same structure.
- **Name Equivalence:** Two types are equivalent only if they have the same name.

**Type Coercion:** Implicit type conversion (widening: int → float).
**Type Casting:** Explicit type conversion.

**Overloading:** Same name for different functions; resolved by type context.
**Polymorphism:** Functions that work on multiple types (e.g., generics).

## 8. Run-Time System

### 8.1 Storage Organization

**Memory Layout (typical):**
```
High Address
┌─────────────────┐
│   Stack         │ ← grows downward; activation records
├─────────────────┤
│   (gap)         │
├─────────────────┤
│   Heap          │ ← grows upward; dynamic allocation (malloc/new)
├─────────────────┤
│   BSS Segment   │ ← uninitialised static data
├─────────────────┤
│   Data Segment  │ ← initialised global and static variables
├─────────────────┤
│   Text Segment  │ ← executable code (read-only)
└─────────────────┘
Low Address
```

### 8.2 Activation Tree and Records

**Activation:** Each call to a procedure creates an activation; these form an **activation tree**.
- Root = main program; children = procedures called by parent.
- At any point, the **active activations** form a path from root to current node.

**Activation Record (Stack Frame):** Memory block allocated on the stack for each procedure invocation.

**Contents of Activation Record:**
```
┌────────────────────────────────┐
│  Return Value                  │
├────────────────────────────────┤
│  Actual Parameters             │
├────────────────────────────────┤
│  Control Link (Dynamic Link)   │ → points to caller's activation record
├────────────────────────────────┤
│  Access Link (Static Link)     │ → points to activation of lexically enclosing scope
├────────────────────────────────┤
│  Saved Machine State           │ (registers, program counter)
├────────────────────────────────┤
│  Local Variables               │
├────────────────────────────────┤
│  Temporaries                   │
└────────────────────────────────┘
```

**Stack Pointer (SP):** Points to top of stack.
**Frame Pointer (FP/BP):** Points to base of current activation record; used for stable addressing.

### 8.3 Parameter Passing Mechanisms

| Mechanism | Description | Language Examples |
|---|---|---|
| **Call by Value** | Copy of actual parameter passed; changes don't affect caller | C, Java (primitives) |
| **Call by Reference** | Address of actual parameter passed; changes affect caller | C++ (&), Fortran |
| **Call by Value-Result** | Copy in; copy out (changes written back on return) | Ada |
| **Call by Name** | Textual substitution of argument (lazy); re-evaluated each use | Algol 60 |
| **Call by Need (Lazy)** | Like call by name but evaluated at most once; result cached | Haskell |

### 8.4 Symbol Table

- Data structure storing information about identifiers in the source program.
- **Attributes stored:** Name, type, scope, size, memory location, dimension (arrays), number of arguments (functions).
- **Operations:** Insert, lookup, delete.

**Implementation:**
- **Linear list:** Simple; O(n) lookup.
- **Hash Table:** O(1) average lookup; preferred for compilers.
- **Binary Search Tree:** O(log n) lookup.

**Scope Management:**
- Each scope (block) has its own symbol table.
- Tables linked in a chain (inner to outer); lookup traverses chain.
- **Open Addressing / Chaining** in hash table.

## 9. Intermediate Code Generation

### 9.1 Intermediate Representations (IR)

**Purpose:** Machine-independent code; bridges front-end and back-end; enables optimisation.

**Types of IR:**

| IR | Description | Example |
|---|---|---|
| **Three-Address Code (TAC)** | At most one operator per instruction; uses temporaries | `t1 = a + b` |
| **Quadruples** | (operator, arg1, arg2, result) | `(+, a, b, t1)` |
| **Triples** | (operator, arg1, arg2) — result is implicit (position in array) | `(+, a, b)` at position i |
| **Indirect Triples** | Pointer array to triples; allows reordering | — |
| **DAG (Directed Acyclic Graph)** | Graph-based; identifies common subexpressions | — |
| **SSA (Static Single Assignment)** | Each variable assigned exactly once; φ-functions for merges | — |
| **P-code** | Stack-based intermediate code (Pascal) | — |

**Three-Address Code Forms:**
```
x = y op z       (binary operation)
x = op y         (unary operation)
x = y            (copy)
goto L           (unconditional jump)
if x relop y goto L    (conditional jump)
param x          (parameter)
call p, n        (procedure call, n = # params)
x = call p, n    (function call with return)
x = y[i]         (indexed read)
x[i] = y         (indexed write)
x = &y, x = *y, *x = y  (pointer operations)
```

### 9.2 Translation of Declarations

- Process declaration of variables; determine types and sizes; allocate offsets.
- **Offset:** Starting from 0 (or after previous variable); width of type determines offset increment.
  - `int` = 4 bytes, `float` = 8 bytes, `double` = 8 bytes, `char` = 1 byte.
- Arrays: `offset(A[i]) = base_addr + i × elem_size`.
- Structures: Lay out fields sequentially (with alignment padding).

### 9.3 Translation of Assignments

- **L-value vs R-value:**
  - L-value: Memory location (address).
  - R-value: Value at that location.
- `x = a + b * c` → TAC:
  ```
  t1 = b * c
  t2 = a + t1
  x  = t2
  ```
- **Array Access:** `x = a[i]` → `t1 = i * width; t2 = a[t1]; x = t2`.
- **Pointer:** `x = *p` → `x = *p`; `*p = x` → `*p = x`.

### 9.4 Translation of Control Flow

**Boolean Short-Circuit Evaluation (backpatching):**
- `A and B`: If A false → skip B (jump to false).
- `A or B`: If A true → skip B (jump to true).
- **Backpatching:** Generate jumps with placeholder labels; fill in when targets are known.
  - `truelist`: List of instruction indices that jump to true label.
  - `falselist`: List of instruction indices that jump to false label.
  - `merge(l₁, l₂)`: Concatenate two lists.
  - `backpatch(list, label)`: Fill in all jumps in list with label.

**If-Else Translation:**
```
if E goto L_true
  goto L_false
L_true: code for S1
  goto L_next
L_false: code for S2
L_next: ...
```

**While Loop:**
```
L_begin: (evaluate E)
  if E goto L_body
  goto L_next
L_body: code for S
  goto L_begin
L_next: ...
```

### 9.5 Translation of Boolean Expressions

- Two methods: Numerical representation (1=true, 0=false) and control-flow method.
- Control-flow method generates conditional jumps directly (more efficient).

### 9.6 Translation of Procedure Calls

```
Caller:
  param a₁
  param a₂
  ...
  param aₙ
  call p, n

Callee:
  (access actual params via AR)
  ...
  return
```

## 10. Code Generation and Code Optimization

### 10.1 Code Generation

**Target Code Forms:** Absolute machine code, relocatable machine code, assembly code.

**Basic Block:** A maximal sequence of three-address instructions where control enters only at the first instruction and leaves only at the last (no internal jumps in or out).

**Basic Block Identification:**
1. First instruction is a leader.
2. Target of a conditional/unconditional jump is a leader.
3. Instruction following a jump is a leader.
4. Each leader and all instructions until the next leader form a basic block.

**Control Flow Graph (CFG):**
- Nodes = basic blocks.
- Edges = control flow between blocks (sequential, branch).

**Register Allocation:** Assign variables to machine registers.
- **Graph Coloring:** Interference graph; two variables interfere if both live at the same point; color with k registers; if graph k-colorable → no spills needed.

### 10.2 Data-Flow Analysis

**Live Variable Analysis:**
- Variable x is **live** at point p if there exists a path from p to a use of x with no intervening definition.
- `use[B]` = variables used in B before definition; `def[B]` = variables defined in B before use.
- **Equations:**
  - `out[B] = ∪ in[S]` for each successor S of B.
  - `in[B] = use[B] ∪ (out[B] − def[B])`
- Solved iteratively (backward pass) until no change (fixed point).

**Reaching Definitions:**
- A definition d reaches point p if there is a path from d to p with no other definition of the same variable.
- `gen[B]` = definitions generated in B; `kill[B]` = definitions killed by B.
- **Equations:**
  - `out[B] = gen[B] ∪ (in[B] − kill[B])`
  - `in[B] = ∪ out[P]` for each predecessor P of B.
- Solved iteratively (forward pass).

**Available Expressions:**
- Expression x+y is available at point p if every path from entry to p evaluates x+y and x, y not redefined after last evaluation.
- Forward analysis; `∩` (intersection) at merge points (conservative approximation — must analysis).

**Summary of Data-Flow Frameworks:**

| Analysis | Direction | Confluence | Initialisation |
|---|---|---|---|
| Live Variables | Backward | Union | out[exit] = ∅ |
| Reaching Definitions | Forward | Union | in[entry] = ∅ |
| Available Expressions | Forward | Intersection | in[entry] = ∅, out[others] = universal set |
| Very Busy Expressions | Backward | Intersection | — |

### 10.3 Local Optimization

Applied within a single basic block:

| Optimization | Description |
|---|---|
| **Common Subexpression Elimination (CSE)** | If `t = a+b` computed earlier and a, b unchanged → reuse t |
| **Dead Code Elimination** | Remove assignments whose values are never used (x not live after) |
| **Constant Folding** | Evaluate constant expressions at compile time: `2 + 3 → 5` |
| **Constant Propagation** | Replace variable known to be constant with its value |
| **Copy Propagation** | If `y = x` and x not redefined, replace uses of y with x |
| **Algebraic Simplification** | `x + 0 = x`, `x * 1 = x`, `x * 0 = 0`, `x * 2 = x + x or x << 1` |
| **Strength Reduction (local)** | Replace expensive operations: `x * 2 → x + x`, `x^2 → x * x` |

### 10.4 Global Optimization

Applied across multiple basic blocks using data-flow analysis:

- **Global CSE:** Using available expressions analysis; eliminate redundant computations across blocks.
- **Global Dead Code Elimination:** Using live variable analysis; remove assignments to variables not live after.
- **Global Constant Propagation:** Constant folding across block boundaries.
- **Copy Propagation:** Across blocks using reaching definitions.

### 10.5 Loop Optimization

**Natural Loop:** A loop with a single entry point (header) and a back edge (edge from node inside loop to header).

**Dominator:** Node d dominates node n if every path from entry to n passes through d.
- **Immediate Dominator (idom):** Closest dominator of n.
- **Dominator Tree:** Captures dominator relationships.

**Back Edge:** Edge `n → d` where d dominates n; identifies natural loops.

**Loop Optimizations:**

| Optimization | Description |
|---|---|
| **Loop-Invariant Code Motion (LICM)** | Move computations whose operands don't change inside loop to pre-header |
| **Induction Variable Elimination** | Identify induction variables (change by fixed amount each iteration); simplify or eliminate |
| **Strength Reduction** | Replace `i * c` in loop with running sum: `s = s + c` each iteration |
| **Loop Unrolling** | Replicate loop body k times; reduce loop overhead; expose instruction-level parallelism |
| **Loop Fusion** | Combine two adjacent loops with same iteration range into one |
| **Loop Fission (Distribution)** | Split one loop into multiple; improve cache performance |
| **Loop Interchange** | Swap inner and outer loops; improve data locality/cache behaviour |

**Induction Variable:** Variable `i` such that `i = i + c` (c constant) on each iteration.
- **Primary induction variable:** Loop counter.
- **Secondary induction variable:** `j = c₁ × i + c₂`; can replace with `j = j + c₁` (strength reduction).

### 10.6 Peephole Optimization

- Applied to a small sliding window of instructions in the target code.
- Looks for local patterns that can be replaced with shorter/faster sequences.

**Types of Peephole Optimizations:**

| Type | Example |
|---|---|
| **Redundant load/store** | `MOV R0, x; MOV x, R0` → Delete second instruction |
| **Unreachable code** | `GOTO L1; ...code...; L1:` → Delete code between goto and L1 |
| **Algebraic simplification** | `ADD R0, #0` → Delete; `MUL R0, #1` → Delete |
| **Strength reduction** | `MUL R0, #2` → `SHL R0, 1` (shift left) |
| **Jump-to-jump** | `GOTO L1; L1: GOTO L2` → Replace with `GOTO L2` |
| **Unnecessary jumps** | `GOTO L; L:` → Delete the goto |

### 10.7 Instruction Scheduling

- Reorder instructions to avoid pipeline stalls and utilise functional units.
- Respects **data dependencies** (RAW, WAR, WAW) and **control dependencies**.

**Data Dependences:**
- **RAW (Read After Write / True Dependence):** `x = a + b; y = x + c` — y reads x after it's written.
- **WAR (Write After Read / Anti-Dependence):** `y = x + c; x = a + b` — x read before it's overwritten.
- **WAW (Write After Write / Output Dependence):** `x = a; x = b` — same variable written twice.

**List Scheduling Algorithm:**
1. Build a data-dependence DAG for the basic block.
2. Assign priorities (e.g., height in DAG = length of longest path to end).
3. Repeatedly select the highest-priority instruction whose predecessors are all scheduled.

**Techniques:**
- **Software Pipelining:** Overlap iterations of a loop; start next iteration before current completes.
- **Modulo Scheduling:** Systematic software pipelining; assigns instructions to slots using initiation interval II.
